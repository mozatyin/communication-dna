## 1. 测试目标

验证 PRD Generator 能否从**纯对话输入**生成**可用于构建游戏的 PRD 文档**。

测试前提：假设读者（和开发者）**完全不知道 Flappy Bird 是什么**。用户从零描述游戏概念，我们检验 PRD 输出是否包含足够的信息让一个开发团队把游戏做出来。

---

## 2. 测试方法

### 2.1 模拟对话设计

编写了 19 条对话消息（10 轮 user + 9 轮 host），模拟用户逐步描述游戏设计：

| 轮次 | 用户描述的内容 |
|------|---------------|
| 1 | 游戏概念：小鸟飞行，躲避障碍 |
| 2 | 核心操作：自动右飞，点击上跳，重力下坠 |
| 3 | 障碍物：成对管子，中间缺口，位置随机 |
| 4 | 难度设计：固定间距，初版固定难度 |
| 5 | 失败条件：碰撞即死，计分=过管数 |
| 6 | 记录系统：历史最高分 + NEW 标志 |
| 7 | 美术风格：像素风、蓝天白云、鲜艳配色 |
| 8 | 动画细节：上仰/下倾/扇翅/死亡翻转 |
| 9 | 音效设计：扇翅/得分/碰撞/落地四种音效 |
| 10 | 界面流程：开始画面→游戏→结束画面 |

同时提供了 16 条结构化 facts 作为辅助输入。

### 2.2 处理流程

```
对话历史 (19条)
    ↓
截断处理 (≤40条, ≤50000字符)
    ↓
IntentionGraph 分析 (Connect → Expand, skip Clarify)
    → 提取 15 个意图节点 + 核心意图
    ↓
组装 LLM Prompt (对话 + Facts + IG上下文 + 语言指令)
    ↓
Claude Sonnet 生成 (temperature=0.7, max_tokens=8000)
    ↓
解析 <prd> 和 <summary> 标签
    ↓
返回 {prd_document, prd_summary, metadata}
```

### 2.3 复现方式

```bash
cd /Users/michael/communication-dna
ANTHROPIC_API_KEY=<your-openrouter-key> .venv/bin/python demo_prd_flappy_scratch.py
```

### 3.4 PRD 额外输出（超出用户描述的部分）

PRD 还补充了以下开发者需要但用户没说的内容：

- **占位策略**：建议先用几何形状做占位符（黄色圆形=小鸟，绿色矩形=管道）
- **打击感细节**：穿越管道时轻微屏幕震动，碰撞时短暂停顿，破纪录时彩带效果
- **背景音乐方向**：轻快循环，节奏匹配游戏操作
- **UI字体建议**：大号白色像素字体，按钮有按下反馈

这些补充是合理的设计建议，不与用户意图冲突。

## 6. 结论

| 维度 | 评估 |
|------|------|
| **结构完整性** | PRD 严格遵守 4 章节格式，下游模块（InterfacePlanGenerator 等）可直接消费 |
| **信息完整性** | 用户描述的 15 个设计要素全部覆盖，无遗漏 |
| **可构建性** | 一个不知道 Flappy Bird 的开发团队拿到这份 PRD 可以实现游戏 |
| **语言一致性** | 中文对话→中文 PRD，符合要求 |
| **IG 集成** | IntentionGraph 正常工作，提供了意图上下文，增强了 PRD 的目标聚焦 |
| **容错性** | 即使 IG 提取失败，PRD 仍可基于对话+facts 生成（本次未触发此分支） |

### 建议技术团队确认的事项

1. **复现测试**：运行 `demo_prd_flappy_scratch.py`，确认输出一致性（LLM 输出有随机性，但结构和覆盖度应稳定）
2. **边界测试**：尝试更短的对话（3-5 轮），观察 `[INFERRED]` 标签是否正确出现在未提及的系统上
3. **英文测试**：运行 `python demo_prd.py flappy`，确认英文对话生成英文 PRD
4. **集成验证**：确认 PRD 输出格式与下游模块（InterfacePlanGenerator / AssetTableGenerator）的解析兼容
5. **IG 失败容错**：模拟 IG 异常（如传入极短对话），确认 `ig_available: false` 时仍能生成可用 PRD
